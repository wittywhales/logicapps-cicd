# Connector Architecture — How Logic App Actions Reach the Backend

## What a connector is

A connector is a managed, Microsoft-hosted HTTP proxy that sits between a Logic App
action and a backend platform (Log Analytics, Exchange Online, Service Bus, etc.).
It handles three things the workflow itself should not own:

- **Protocol translation** — the workflow speaks a generic JSON action schema;
  the connector translates that into the backend's actual REST API calls
- **Authentication brokering** — the workflow authenticates to the connector;
  the connector separately authenticates to the backend on its behalf
- **Schema / swagger** — the connector publishes an OpenAPI spec that the
  designer uses to render the action inputs and outputs

The workflow never calls `api.loganalytics.io` or `graph.microsoft.com` directly.
It always calls the connector, and the connector calls the platform.

## The connection runtime URL

The runtime URL is the address of the connector instance inside the API hub.
It is unique per connection resource. Format:

```
https://<envId>.<scaleUnit>.common.logic-<region>.azure-apihub.net/apim/<connector>/<connectionInstanceId>
```

From this project:
```
https://436bb970c38a8c3d.11.common.logic-westeurope.azure-apihub.net/apim/azuremonitorlogs/66c127e7a0a94e559cd778c6fb366610
```

| Segment | Value | Meaning |
|---|---|---|
| `436bb970c38a8c3d` | environment ID | Ties back to `x-ms-environment-id` in response headers. Identifies the Logic App's hosting environment. |
| `11` | scale unit | The API hub partition serving this region. |
| `common` | tenancy model | Shared multi-tenant hub (as opposed to isolated). |
| `logic-westeurope` | region | Must match the Logic App's region. |
| `azure-apihub.net` | hub domain | The Microsoft-managed API proxy layer. |
| `azuremonitorlogs` | connector name | Which managed API this connection belongs to. |
| `66c127e7a0a94e559cd778c6fb366610` | connection instance ID | Unique ID of this specific connection resource. |

The URL is **generated by Azure when the connection resource is created** and does
not change unless the connection is deleted and recreated. It is read from the
connection resource via:

```bash
az rest --method get \
  --url "https://management.azure.com<CONNECTION_RESOURCE_ID>?api-version=2016-06-01" \
  --query "properties.connectionRuntimeUrl" -o tsv
```

This is why `AZUREMONITORLOGS_CONNECTION_RUNTIMEURL` is stored as an app setting
and excluded from Terraform state — it is a stable runtime artefact, not
something Terraform creates or controls.

## Authentication at each hop

There are two separate auth relationships, and they use different tokens.

```
Logic App  ──── hop 1 ────▶  API Hub  ──── hop 2 ────▶  Backend API
           MSI token                    OBO token
           aud=management              aud=api.loganalytics.io
           .core.windows.net           (or graph.microsoft.com, etc.)
```

### Hop 1 — Logic App → API Hub

Controlled by the top-level `authentication` block in `connections.json`:

```json
"authentication": {
  "type": "ManagedServiceIdentity"
}
```

The Logic App runtime acquires an MSI token for the audience defined in
`connectionProperties.authentication.audience` and sends it in the
`Authorization: Bearer` header to the connection runtime URL.

The API hub validates this token and checks it against the **access policy**
on the connection resource (created in `connections.tf`). The access policy
says: *this principal is allowed to use this connection*.

### Hop 2 — API Hub → Backend

Controlled by `connectionProperties.authentication` in `connections.json`:

```json
"connectionProperties": {
  "authentication": {
    "audience": "https://management.core.windows.net/",
    "additionalAudiences": ["https://api.loganalytics.io"],
    "type": "ManagedServiceIdentity"
  }
}
```

The API hub performs an OBO (On-Behalf-Of) exchange: using the inbound MSI
token, it acquires a new token for each audience in `additionalAudiences` and
forwards the request to the backend with that token.

`audience` is the primary resource. `additionalAudiences` lists extra resources
the connector calls internally. For `azuremonitorlogs`, the connector's
management-plane operations use `management.core.windows.net`, but the actual
KQL query goes to `api.loganalytics.io`, so both are needed.

### What breaks when these are wrong

| Misconfiguration | Symptom |
|---|---|
| `connectionRuntimeUrl` is empty | `404` or request never reaches the API hub |
| Access policy missing | `401 Unauthorized` from the API hub |
| Wrong `audience` in `connectionProperties` | API hub gets a token for the wrong resource; backend rejects it |
| `additionalAudiences` missing | API hub falls back to OAuth consent endpoint; fails with `Token not found` |
| MSI has no RBAC role on the backend | Token is valid but backend returns `403 Forbidden` |

## Sequence for an IngestionSpike query action

```
Logic App runtime
  │
  │  reads connections.json
  │  resolves "azuremonitorlogs" → connectionRuntimeUrl + auth config
  │
  │  acquires MSI token
  │    audience = https://management.core.windows.net/
  │    identity = la-ingestionspike-local (67a8db6f-...)
  │
  ├─▶ POST .../apim/azuremonitorlogs/66c127e7.../queryData
  │     Authorization: Bearer <MSI token>
  │     Body: { query: "Usage | ...", subscriptions: "...", workspaceName: "epamlaw" }
  │
API Hub (azure-apihub.net)
  │
  │  validates MSI token
  │  checks access policy → principal 67a8db6f is allowed
  │
  │  reads connectionProperties.authentication
  │  performs OBO: acquires token for api.loganalytics.io
  │    on behalf of principal 67a8db6f
  │
  ├─▶ POST https://api.loganalytics.io/v1/workspaces/epamlaw/query
  │     Authorization: Bearer <OBO token, aud=api.loganalytics.io>
  │     Body: { query: "Usage | ..." }
  │
Log Analytics workspace
  │
  │  validates token
  │  checks RBAC → 67a8db6f has Log Analytics Reader on workspace epamlaw ✓
  │  executes KQL
  │
  └─▶ { "tables": [...] }   ← flows back through API Hub → Logic App runtime
```

## How office365 differs

The `office365` connector follows the same proxy architecture but uses delegated
user auth instead of MSI for hop 2.

```
Logic App  ──── hop 1 ────▶  API Hub  ──── hop 2 ────▶  graph.microsoft.com
           MSI token                    OAuth user token
           (access policy)             stored in connection
                                       (set during portal consent)
```

When a user authorizes the connection in the portal, their OAuth token (with
refresh token) is stored in the connection resource by Azure. On each workflow
run, the API hub retrieves the stored token, refreshes it if needed, and calls
Graph on behalf of that user. This is why the email appears to come from a real
mailbox — it is sent as that user.

Because hop 2 requires a stored user token (not an MSI OBO), there is no way
to fully automate the initial setup. The `OFFICE365_CONNECTION_RUNTIMEURL` is
available immediately after connection creation, but the connection will not
function until consent is completed.

## Where each config piece lives and why

| Config | Location | Controls |
|---|---|---|
| Connection resource (`kind: V2`, `parameterValueSet`) | `connections.tf` | Creates the connector instance; declares that MSI auth is used for the connection |
| Access policy | `connections.tf` | Authorises the Logic App's MSI to use the connection (hop 1 gate) |
| `connectionRuntimeUrl` app setting | set by pipeline, excluded from Terraform | The address the Logic App calls; stable after connection creation |
| `connectionProperties.authentication` | `connections.json` | Tells the API hub what token(s) to acquire for the backend (hop 2) |
| Top-level `authentication` | `connections.json` | Tells the Logic App runtime how to authenticate to the API hub (hop 1) |
| RBAC role assignment | `main.tf` | Grants the MSI permission on the backend resource (hop 2 authorisation) |
