# Logic App Standard — IaC Management Guide

## Repository Structure

```
logicapps/
├── infrastructure/          # Terraform — all Azure resources
│   ├── main.tf              # Logic App, storage, plan, observability, RBAC
│   ├── connections.tf       # API connections + access policies (ARM)
│   ├── variables.tf
│   ├── outputs.tf
│   └── environments/
│       ├── dev.tfvars
│       ├── prod.tfvars
│       └── local.tfvars
├── workflows/               # Logic App content — deployed via zip
│   ├── connections.json     # Connection references (reads from app settings)
│   ├── parameters.json      # Workflow parameters (reads from app settings)
│   ├── host.json
│   ├── IngestionSpike/
│   │   └── workflow.json
│   └── HealthCheck/
│       └── workflow.json
└── pipelines/
    ├── azure-pipelines.yml       # Full pipeline: infra + workflows
    ├── infra-only-pipeline.yml   # Infrastructure changes only
    └── templates/
        ├── terraform-steps.yml   # Init → Plan → Publish artifact
        └── deploy-workflows.yml  # Sync runtime URLs → zip deploy
```

## What Is Fully Automated

### Terraform (`infrastructure/`)

| Resource | File | Notes |
|---|---|---|
| Resource group | `main.tf` | |
| Storage account | `main.tf` | Required by Logic App Standard runtime |
| App Service Plan (WS1) | `main.tf` | |
| Logic App Standard | `main.tf` | System-assigned managed identity enabled |
| Log Analytics workspace | `main.tf` | For Logic App diagnostics |
| Application Insights | `main.tf` | Wired to the above workspace |
| `azuremonitorlogs` API connection | `connections.tf` | Created via ARM, `managedIdentityAuth` parameter set |
| `office365` API connection | `connections.tf` | Created via ARM, awaiting OAuth consent |
| Access policies on both connections | `connections.tf` | Grants the Logic App's MSI permission to call each connection at runtime |
| Log Analytics Reader role assignment | `main.tf` | Scoped to the target workspace; enables the MSI to query it |
| All app settings | `main.tf` | Connection IDs, API IDs, workspace config, email recipients |

### Pipeline — Stage 3: Deploy Workflows (`deploy-workflows.yml`)

| Step | What it does |
|---|---|
| **Sync runtime URLs** | Reads `connectionRuntimeUrl` from each connection resource via `az rest`, writes them back to the Logic App's app settings |
| **Package workflows** | Zips the `workflows/` directory (excludes local dev artefacts) |
| **Zip deploy** | Pushes the zip to the Logic App via the Functions zip deploy API; restarts the app |

The workflow files (`connections.json`, `parameters.json`, `workflow.json`) never contain hardcoded values — everything is read from app settings via `@appsetting('...')` at runtime.

## What Requires Manual Action

### 1. Office 365 OAuth Consent — one-time per environment

The `office365` connection uses **delegated user auth**, not managed identity. A user must authorize it once after the connection is first created.

**Steps:**
1. Go to Azure Portal → Resource Group → `<name>-office365-v2` connection
2. Click **Edit API connection** → **Authorize**
3. Sign in with the mailbox account that will send alerts
4. Click **Save**

> The `OFFICE365_CONNECTION_RUNTIMEURL` is set by Terraform immediately after connection creation — no manual action is needed to get the URL. Portal authorization is required only to make the connection actually send email.

### 2. `OFFICE365_CONNECTION_RUNTIMEURL`

This setting is set by Terraform (`connections.tf` ARM output → `main.tf` app setting) at deploy time. The URL is generated by Azure when the connection resource is created, regardless of auth state.

The connection will not send email until a user completes OAuth consent (step 1 above), but the URL itself is stable and correct from first apply. No pipeline step or manual `az cli` is needed to sync it.

---

## Connection Runtime URL Behaviour

Both `azuremonitorlogs` and `office365` runtime URLs are:
- **Generated by Azure at connection creation** — available immediately in the ARM deployment output
- **Read by Terraform** from `connections.tf` ARM output and passed to `main.tf` as app settings
- **Stable** — they do not change unless the connection resource is destroyed and recreated

The pipeline no longer needs to sync runtime URLs.

## Deployment Lifecycle

### First Deploy (new environment)

```
1. terraform apply          → creates all Azure resources
                               connections created, runtime URLs set from Terraform output
                               MSI has Log Analytics Reader role
                               access policies exist on both connections

2. Authorize Office 365     → MANUAL: portal OAuth consent
   connection               → office365 connection becomes functional for sending email
                               (the runtime URL was already set in step 1)

3. Workflow is live
   (azure monitor logs works immediately after step 1;
    office365 email sending works after step 2)
```

### Subsequent Deploys — Workflow Change Only

Push to `main` → pipeline triggers automatically (path filter: `Terraform/logicapps/**`):

```
Stage 1: Terraform Plan     → typically no-op if infra unchanged
Stage 2: Terraform Apply    → applies only diffs; ignores runtime URL settings
Stage 3: Deploy Workflows   → re-syncs runtime URLs (idempotent)
                              → zip deploys updated workflow content
```

### Subsequent Deploys — Infrastructure Change Only

Use `infra-only-pipeline.yml` (manual trigger). Does not touch workflow content.

```
Stage 1: Terraform Plan
Stage 2: Terraform Apply
```

## Key Configuration Files

### `connections.json`
Maps workflow connection names to app settings. The `connectionProperties.authentication` block controls how the API hub acquires tokens for each backend:

```json
"azuremonitorlogs": {
  "connectionProperties": {
    "authentication": {
      "additionalAudiences": ["https://api.loganalytics.io"],
      "audience": "https://management.core.windows.net/",
      "type": "ManagedServiceIdentity"
    }
  },
  "authentication": { "type": "ManagedServiceIdentity" }
}
```

- `audience` — token the Logic App sends to the connection gateway
- `additionalAudiences` — tells the gateway it also needs a token for `api.loganalytics.io`; without this, it falls back to OAuth consent and fails

### `parameters.json`
Maps workflow parameter names to app settings. All target workspace coordinates and notification config come from here — no values are hardcoded in the workflow JSON itself.

### `environments/*.tfvars`
One file per environment. Contains the target Log Analytics workspace, email recipients, and resource naming. The only file that changes between dev/prod.

## Adding a New Workflow

1. Create `workflows/<WorkflowName>/workflow.json`
2. If it uses a new connection type, add the connection resource to `connections.tf`, add its app settings to `main.tf`, and add its entry to `connections.json`
3. Push to `main` — the pipeline handles the rest

No Terraform changes are needed for a workflow that uses existing connections.

## Common Operations Reference

| Operation | How |
|---|---|
| Add/change a workflow | Edit `workflows/`, push to `main` |
| Change alert recipients | Update `alert_email_recipients` in `environments/<env>.tfvars`, push |
| Change target workspace | Update `la_target_*` vars in `environments/<env>.tfvars`, push |
| Re-authorize Office 365 | Portal → connection → Authorize, then re-run Stage 3 |
| Manual workflow trigger | `az rest --method post --url ".../triggers/<trigger>/run?api-version=2018-11-01"` |
| Infra change only | Trigger `infra-only-pipeline.yml` manually |
| New environment | Copy `environments/dev.tfvars`, set `environment` variable in pipeline |
